Scripts vs Plugins

Welcome! RealmEngine lets you extend the client with your own code. There are two ways to do that — scripts and plugins — and they're built for different kinds of tasks. This page helps you pick the right one before you start coding.

Every guide in this section starts from scratch and assumes no prior RealmEngine experience. Basic JavaScript knowledge helps, but you can copy-paste your way through the examples and pick things up as you go.

At a Glance

Both live in your Documents folder and both are written in JavaScript (or TypeScript, if you want editor autocomplete and type checks).

Documents/
└── Realmengine/
    ├── Scripts/   ← automation sequences — "do a task, then finish"
    └── Plugins/   ← persistent behaviors — "always on in the background"

Scripts

A script is a program you start and stop like an app. It runs a loop until you hit stop or it finishes its job. Typical uses:

  • Farm a specific dungeon over and over
  • Walk a character through a quest path
  • Level up until you hit 20, then stop
  • Re-roll a stat until you get the value you want

Technical shape:

  • Lives in its own folder with a realmengine.script.json manifest (name, developer, version, entry file).
  • Default-exports a class with three methods: onStart(), onLoop(), onStop().
  • Uses the typed SDK: RealmEngine.chat, RealmEngine.inventory, and so on — full listing in the API Reference tab.
  • Shows up in the Scripts tab. Only one runs at a time.
  • You either write it locally or install it from the marketplace (where it's reviewed by us).

Plugins

A plugin is a piece of code that reacts to things happening in the game. It doesn't have a main task or a stop condition — once you flip the toggle on, it sits in the background and listens. Typical uses:

  • Filter out chat messages that match a blocklist
  • Show a damage log in the HUD
  • Add a custom /command only you can see
  • Auto-nexus when your HP drops below a threshold

Technical shape:

  • A single .mjs file — no manifest, no folder, no build step.
  • Exports one function: register(ctx). The ctx argument is an object full of hooks you can attach handlers to.
  • Shows up in the Plugins tab with a toggle switch. You can have many enabled at once.
  • Community plugins aren't reviewed — see Trust Model below.

Which Should You Build?

Ask yourself a single question: does this task have a natural ending?

  • Yes — it runs through a sequence and stops: write a script.
  • No — it should always be listening in the background: write a plugin.

A few examples to calibrate:

  • "Farm Tomb of the Ancients 10 times" → script (has a stop condition)
  • "Play a sound when someone whispers me" → plugin (always listening)
  • "Level up a fresh character to 20" → script (done once you hit the level)
  • "Highlight low-HP enemies on screen" → plugin (always on while playing)

You can have both. They share the same RealmEngine SDK where it makes sense, so skills transfer.

Trust Model

Scripts installed from the RealmEngine marketplace are reviewed by us before they're published. Community plugins are not reviewed — they're JavaScript files the community shares freely, and they run on your machine. Their reach into the game is limited to what the SDK exposes (raw packet access, the internal proxy bus, and low-level game-data internals are reserved for bundled first-party plugins), but they can still drive anything the SDK can do — chat, movement, combat, auto-actions — and they run with whatever permissions Node.js has on your computer. Only load community plugins from people you trust, and stick to the bundled plugins (the ones that ship with the client) if you're not sure.

Creating Your First Script

This guide walks you through building the smallest possible working script — a file that prints a message every second. By the end you'll know exactly what goes where and be ready to write something real.

A script in RealmEngine is a folder. The folder has a manifest that tells the client what to call the script, who made it, and which file to run. That file exports a class with three lifecycle methods. That's the whole model.

Where Scripts Live

RealmEngine scans one folder on your computer:

Documents/
└── Realmengine/
    └── Scripts/   ← drop each script here as its own folder

If Documents/Realmengine/Scripts/ doesn't exist yet, open RealmEngine once — it's created automatically on first launch.

Folder Layout

Every script lives in its own sub-folder. Here's what a fully-loaded one looks like:

Documents/
└── Realmengine/
    └── Scripts/
        └── my-script/            ← the folder name is yours to pick
            ├── realmengine.script.json  ← required, the manifest
            ├── src/                 ← TypeScript source (optional)
            │   └── index.ts
            ├── dist/                ← built output RealmEngine runs
            │   └── index.mjs
            ├── package.json
            ├── tsup.config.ts
            └── tsconfig.json

Only the manifest and the entry file (the one the manifest points at) are strictly required. The rest are there because most real scripts are written in TypeScript and bundled with TSUP — covered in the next guide. For this first script we'll write the entry file by hand so there's nothing to build.

Step 1: Write the Manifest

Create a file called realmengine.script.json inside your script folder and paste in:

{
  "name": "My Script",
  "developer": "YourName",
  "version": "1.0.0",
  "entry": "dist/index.mjs"
}
  • name — what shows up in the Scripts tab.
  • developer — your handle, displayed next to the name.
  • version — any version string, e.g. "1.0.0". RealmEngine shows it but doesn't enforce a format.
  • entry — the file RealmEngine actually loads, relative to the script folder. It must end in .mjs.

If the manifest is missing or any field is blank, RealmEngine skips the folder entirely.

Step 2: Write the Entry File

Create dist/index.mjs (make the dist folder too) and paste in the example below. This is plain JavaScript — no build step needed:

export default class MyScript {
  onStart() {
    console.log('[MyScript] started');
  }

  onLoop() {
    console.log('[MyScript] tick');
    return 1000; // call me again in 1 second
  }

  onStop() {
    console.log('[MyScript] stopped');
  }
}

Three things are happening here:

  • export default class MyScript { … } tells RealmEngine which class to instantiate. The class name doesn't matter — the default export does.
  • Each method (onStart, onLoop, onStop) is a stage of the lifecycle, explained below.
  • console.log messages show up in the Script log panel inside RealmEngine.

If you'd rather use TypeScript, put your source in src/index.ts, implement the Script type from the SDK, and bundle to dist/index.mjs with TSUP (see the next guide). The runtime shape is identical.

The Lifecycle

RealmEngine runs your class in a fixed order:

onStart() | onLoop() -> wait returned milliseconds -> onLoop() | onStop()
  • onStart() runs once when you press Run. Use it for setup: subscribe to events, reset counters, print a hello message.
  • onLoop() runs over and over. Whatever number you return is how long (in milliseconds) RealmEngine waits before calling it again. Return 1000 for once a second, 250 for four times a second.
  • onStop() runs once when you press Stop, when the script throws an unrecoverable error, or when RealmEngine shuts down. Use it for cleanup.

All three are required — RealmEngine refuses to load a class that's missing any of them, even if you don't need to do anything in one.

Step 3: Run It

  1. Open RealmEngine.
  2. Go to the Scripts tab.
  3. Press Refresh if your script doesn't appear.
  4. Find My Script in the list and press Run.
  5. You should see [MyScript] tick every second in the Script log.
  6. Press Stop when you've seen enough.

Congratulations — that's a complete working script. Everything from here on is learning how to do more inside those three methods.

Doing Useful Work: The SDK

Inside your methods you have access to RealmEngine — the SDK object that exposes the game: chat, inventory, world state, players, and so on. Here's a slightly more useful script that logs every chat message:

import { RealmEngine } from '@realmengine/sdk';

export default class EchoScript {
  onStart() {
    this.unsubscribe = RealmEngine.chat.onMessage((e) => {
      console.log(`${e.sender}: ${e.message}`);
    });
  }

  onLoop() {
    return 2000; // we react to events, so loop slowly
  }

  onStop() {
    this.unsubscribe?.();
  }
}

A few new ideas in this example:

  • RealmEngine.chat.onMessage(handler) registers a listener. It returns an unsubscribe function — calling it detaches the listener.
  • We save that unsubscribe on this.unsubscribe in onStart and call it from onStop. If we didn't, pressing Stop then Run would add a second listener on top of the first and every chat line would print twice.
  • onLoop still has to return a number, even if your work is purely event-driven. A long interval (2000ms here) keeps CPU usage low.

The full list of SDK namespaces and methods is in the API Reference tab — browse it to see what's available (chat, inventory, world, players, packets, and more).

Clean Up in onStop

Every listener, timer, or resource you create in onStart should be torn down in onStop. If your script registers several of them, it's usually cleanest to keep the unsubscribe functions in an array:

import { RealmEngine } from '@realmengine/sdk';
import type { Script, Unsubscribe } from '@realmengine/sdk';

export default class MyScript implements Script {
  private unsubs: Unsubscribe[] = [];

  onStart(): void {
    this.unsubs.push(
      RealmEngine.chat.onMessage((e) => {
        console.log(e.sender, e.message);
      })
    );
  }

  onLoop(): number {
    return 2000;
  }

  onStop(): void {
    this.unsubs.forEach((unsubscribe) => unsubscribe());
    this.unsubs = [];
  }
}

This version is TypeScript — it uses the Script and Unsubscribe types from the SDK for editor autocomplete and compile-time checks. The JavaScript version is the same code without the type annotations and the implements Script bit.

Packaging With TSUP

RealmEngine loads one .mjs file per script. That's fine if your whole script fits in a single file — but once you split your code across multiple TypeScript files you need a bundler to stitch them together. TSUP is the bundler we recommend: it's a thin wrapper around esbuild that "just works" for scripts with almost no configuration.

This guide walks you through setting TSUP up from scratch inside the my-script folder you created in Creating Your First Script. By the end you'll be able to write TypeScript across as many files as you like and produce a single dist/index.mjs that RealmEngine can run.

Do You Actually Need This?

  • Yes if you want to write TypeScript (RealmEngine only runs .mjs, so .ts has to be compiled).
  • Yes if your script is spread across more than one file.
  • Yes if you want to use any npm package beyond the RealmEngine SDK.
  • No if you just have a single hand-written .mjs file with no imports — in that case, skip TSUP and drop the file straight into dist/index.mjs.

Before You Start

  • Install Node.js 20 or newer. This gives you node and npm — npm is what installs TSUP.
  • Open a terminal inside your script folder (e.g. Documents/Realmengine/Scripts/my-script/). All commands below run from here.

Step 1: Create package.json

This file tells npm which packages to install and defines the short commands you'll run (npm run build, npm run watch). Save it at the root of your script folder:

{
  "type": "module",
  "scripts": {
    "build": "tsup",
    "watch": "tsup --watch"
  },
  "dependencies": {
    "@realmengine/sdk": "*"
  },
  "devDependencies": {
    "tsup": "^8.0.0",
    "typescript": "^5.0.0"
  }
}
  • "type": "module" — treat files in this folder as ES modules (the same module system RealmEngine uses).
  • scripts.build — what npm run build actually runs. We shorten npx tsup to tsup.
  • dependencies — packages your script imports at runtime. Only the SDK goes here.
  • devDependencies — tools used only while building. The user never needs these to run your script.

Step 2: Install the Tools

From your script folder, run:

npm install

This downloads TSUP, the TypeScript compiler, and the RealmEngine SDK types into a new node_modules/ folder. Don't edit anything in there by hand — npm manages it for you. You'll also see a package-lock.json appear; commit it alongside package.json if you use git.

Step 3: Create tsup.config.ts

This file tells TSUP what to build and how. Save it alongside package.json:

import { defineConfig } from 'tsup';

export default defineConfig({
  entry: ['src/index.ts'],
  format: ['esm'],
  platform: 'node',
  target: 'node20',
  splitting: false,
  sourcemap: true,
  clean: true,
  dts: false,
  outDir: 'dist',
  external: ['@realmengine/sdk'],
  outExtension() {
    return { js: '.mjs' };
  },
});

Each option in plain English:

  • entry — the one file TSUP starts reading from. Every import it encounters gets pulled into the output.
  • format: ['esm'] — emit ES modules. RealmEngine loads scripts via dynamic import(), which only understands ESM.
  • platform: 'node' + target: 'node20' — modern Node syntax (optional chaining, top-level await, etc.) is left alone rather than polyfilled.
  • external: ['@realmengine/sdk'] — don't copy the SDK into your bundle. RealmEngine injects the live SDK at runtime; bundling a copy would just duplicate it and break types like Script.
  • outDir: 'dist' + outExtension — write the result to dist/index.mjs. This path must match the entry field in your realmengine.script.json manifest.
  • sourcemap: true — also emit dist/index.mjs.map so runtime stack traces point at your original .ts lines instead of bundled output.
  • clean: true — wipe dist/ before each build so stale files can't linger.
  • splitting: false — always produce a single output file. RealmEngine expects exactly one .mjs.
  • dts: false — skip generating .d.ts type declaration files. Scripts don't need them.

Step 4: Write Your Source

Put every TypeScript file you author under src/. Start with src/index.ts — this is the file listed in entry, and it's where TSUP begins bundling.

import { createCombatBranch } from './branches/combat';
import { createVaultBranch } from './branches/vault';
import { RealmEngine } from '@realmengine/sdk';
import type { Script } from '@realmengine/sdk';

export default class MyScript implements Script {
  onStart() { /* … */ }
  onLoop() { return 1000; }
  onStop()  { /* … */ }
}

The local imports (./branches/combat, ./branches/vault) are bundled into dist/index.mjs. The @realmengine/sdk import is left alone — it resolves to the live SDK RealmEngine provides when the script runs.

Step 5: Build

npm run build

TSUP reads your config, compiles every .ts file it reaches from the entry point, and writes the bundle:

my-script/
└── dist/
    ├── index.mjs         ← the file RealmEngine loads
    └── index.mjs.map     ← source map, keep it alongside .mjs

Open RealmEngine, go to the Scripts tab, and your script should now appear ready to run. If it was already running from an older build, stop it and start it again to pick up the new bundle.

Step 6 (Optional): Watch Mode

Re-running npm run build after every edit gets old fast. Open a second terminal and leave this running in the background:

npm run watch

TSUP now rebuilds dist/index.mjs automatically every time you save a source file. RealmEngine doesn't auto-reload scripts — after each rebuild, stop and restart the script from the Scripts tab to pick up the new bundle.

If Something Goes Wrong

  • "Cannot find module 'tsup'" — you skipped npm install. Run it from the script folder.
  • Script doesn't show up in the Scripts tab — your manifest's entry field doesn't match where TSUP actually wrote the file. Both should be dist/index.mjs.
  • Type errors about @realmengine/sdk — make sure it's listed as external in tsup.config.ts. Never bundle the SDK.
  • Edits aren't applying — stop and restart the script in RealmEngine after each rebuild, even in watch mode.
  • Runtime errors point at mangled line numbers — check that dist/index.mjs.map exists next to dist/index.mjs. That's what maps errors back to your TypeScript.

Running Script Packages

Once you've built a script — either by hand or with TSUP — here's exactly how RealmEngine finds it, shows it in the dashboard, and runs it. This guide also covers what to do when it doesn't show up.

Scripts Folder

RealmEngine scans a single folder for scripts: Documents/Realmengine/Scripts/. Each script lives in its own sub-folder directly under that. The scan is one level deep only — don't nest folders inside other folders.

Documents/
└── Realmengine/
    └── Scripts/
        ├── my-script/
        │   ├── realmengine.script.json
        │   └── dist/
        │       └── index.mjs
        └── aio/
            ├── realmengine.script.json
            └── dist/
                └── index.mjs

You can have as many sub-folders as you like. Each one is a separate, independent script.

The Scripts Tab

Steps to go from "I just saved a file" to "it's running in-game":

  1. Open RealmEngine and switch to the Scripts tab.
  2. If your script isn't listed, press the Refresh button at the top of the tab. RealmEngine re-scans the folder and picks up any new or changed manifests.
  3. Your script appears as a row with its name, developer, version, entry path, and current status (Stopped, Running, or Error).
  4. Press Run on the row. RealmEngine imports your entry file, calls onStart(), then starts the onLoop() schedule.
  5. Press Stop to end it. RealmEngine clears the loop timer and calls onStop().
  6. Anything your script prints via console.log appears in the Script log panel below the list.

Only one script can be in the Running state at a time. Starting a second one automatically stops the first.

Entry File Rules

  • The manifest's entry must end in .mjs. .js, .cjs, and .ts are ignored.
  • The path is relative to the script folder and must stay inside it — you can't reach into a sibling or a parent directory.
  • The folder name becomes a stable script id. If you rename the folder, RealmEngine treats it as a brand-new script (any saved state is lost).
  • Loose .js or .mjs files dropped directly under Scripts/ without a manifest folder are ignored.

Reloading After a Code Change

When you edit your script's source and rebuild, RealmEngine doesn't automatically pick up the change. Press Stop, then Run again on the row — each run cache-busts the import so you always get the latest dist/index.mjs.

If you added an entire new script while RealmEngine was open, press Refresh first so the new folder appears in the list.

Structuring With TreeScript

Once your script is doing more than one thing — combat and looting and nexusing and vaulting — a single onLoop() gets messy fast. TreeScript is a tiny framework that lets you organise a script as a decision tree of branches (conditions) and leaves (actions), so each piece of behavior stays small, named, and independent.

It's modelled on DreamBot's TreeScript, so if you're coming from there the shape will look familiar. Everything ships in @realmengine/sdk — there's nothing new to install.

When to Reach for It

You don't have to use TreeScript. A plain Script class with a handwritten onLoop() is fine for small scripts. Reach for TreeScript when one or more of these start to hurt:

  • Your onLoop() is a tower of if/else if branches.
  • You keep forgetting the order the checks need to run in (heal before attack, escape before heal, …).
  • You want a status like "Current task: Combat → Attacking Oryx" without adding a this.currentState field.
  • You want to unit-test a single piece of behavior in isolation.

The Vocabulary

Four classes, arranged in a tree:

  • Leaf — a terminal action. Implements isValid() ("should I run right now?") and onLoop() ("do the work, return sleep ms").
  • Branch — a Leaf that also holds child leaves. Implements isValid() and (by default) runs the first valid child.
  • Root — a built-in Branch that is always valid. You don't make one yourself; TreeScript hands you one.
  • TreeScript — a full RealmEngine Script. You extend it, add branches/leaves in onStart(), and let its built-in onLoop() walk the tree each tick.

The picture:

Root (always valid)
 ├── CombatBranch (isValid: enemies nearby)
 │    ├── NexusLowHp  (isValid: HP < 30%)
 │    └── AttackNearest (isValid: always, inside combat)
 ├── LootBranch (isValid: loot bag nearby)
 │    └── GrabBag (isValid: always, inside loot)
 └── IdleLeaf (isValid: always — the fallback)

How Traversal Works

Every tick, TreeScript walks from the root, one level at a time:

  1. Look at the current branch's children in insertion order.
  2. Call isValid() on each. The first one that returns true wins.
  3. If the winner is a Branch, descend into it and repeat step 1.
  4. If the winner is a Leaf, call its onLoop() and sleep for the ms it returned.
  5. If nothing on the current level is valid, sleep for idleSleep (default 100 ms) and try again next tick.

Two rules fall out of this:

  • Order matters. Put the highest-priority branches first — a combat branch should come before a loot branch so the bot doesn't grab a bag while getting hit.
  • Always end with a catch-all. Put a leaf whose isValid() returns true at the end of the root so the tree never falls through to "nothing valid."

Step 1: Scaffold the Script

Start from the script skeleton in Creating Your First Script and Packaging With TSUP. TreeScript is just another class you extend inside your entry file:

import { TreeScript, RealmEngine } from '@realmengine/sdk';

export default class MyScript extends TreeScript {
  onStart(): void {
    RealmEngine.log.info('tree ready');
    // tree construction goes here
  }

  onStop(): void {
    RealmEngine.log.info('tree stopped');
  }
}

You don't need to write onLoop() — TreeScript already has one that walks the tree. You can override it if you want to do something extra before/after the walk, but it's rarely necessary.

Step 2: Add Your First Leaf

A leaf is any class that extends Leaf and answers two questions — should I run, and what do I do?

import { TreeScript, Leaf, RealmEngine } from '@realmengine/sdk';

class HeartbeatLeaf extends Leaf {
  isValid(): boolean {
    return true; // always runs
  }

  onLoop(): number {
    RealmEngine.log.info(`HP: ${RealmEngine.self.getHP()}`);
    return 2000; // sleep 2s before next tick
  }
}

export default class MyScript extends TreeScript {
  onStart(): void {
    this.addBranches(new HeartbeatLeaf());
  }
}

Press Run and you'll see HP: … log lines every 2 seconds — same as a hand-written onLoop, but expressed as a reusable piece.

Step 3: Add a Branch

Branches gate whole subtrees behind a condition. This one runs combat logic only when enemies are nearby:

import { TreeScript, Branch, Leaf, RealmEngine } from '@realmengine/sdk';

class NexusLowHp extends Leaf {
  isValid(): boolean { return RealmEngine.self.getHPPercent() < 0.3; }
  onLoop(): number {
    RealmEngine.self.escape();
    return 3000;
  }
}

class AttackNearest extends Leaf {
  isValid(): boolean { return RealmEngine.enemies.count() > 0; }
  onLoop(): number {
    const e = RealmEngine.enemies.getNearest();
    if (e) RealmEngine.combat.attack(e.objectId);
    return 150;
  }
}

class CombatBranch extends Branch {
  isValid(): boolean { return RealmEngine.enemies.count() > 0; }
}

class IdleLeaf extends Leaf {
  isValid(): boolean { return true; }
  onLoop(): number { return 500; }
}

export default class MyScript extends TreeScript {
  onStart(): void {
    this.addBranches(
      new CombatBranch().addLeaves(new NexusLowHp(), new AttackNearest()),
      new IdleLeaf(),
    );
  }
}

Each tick the walker enters Root, checks CombatBranch.isValid(). If enemies exist it descends, checks NexusLowHp first (because it's listed first), otherwise AttackNearest. Otherwise it falls through to IdleLeaf. Notice how each class describes exactly one behavior — nothing reaches across pieces.

File / Folder Structure

For anything beyond a handful of leaves, use a branches/ folder. Each branch gets its own sub-folder containing the branch file and a leaves/ folder for everything that runs under it.

my-script/
├── realmengine.script.json
├── package.json
├── tsup.config.ts
├── tsconfig.json
├── src/
│   ├── index.ts                        ← exports default MyScript extends TreeScript
│   ├── branches/
│   │   ├── combat/                    ← one folder per branch
│   │   │   ├── CombatBranch.ts
│   │   │   └── leaves/            ← leaves that live under this branch
│   │   │       ├── NexusLowHp.ts
│   │   │       ├── AttackNearest.ts
│   │   │       └── StrafeDodge.ts
│   │   ├── loot/
│   │   │   ├── LootBranch.ts
│   │   │   └── leaves/
│   │   │       └── GrabBag.ts
│   │   └── vault/
│   │       ├── VaultBranch.ts
│   │       └── leaves/
│   │           └── DepositLoot.ts
│   ├── IdleLeaf.ts                     ← root-level fallback, no branch of its own
│   └── util/                           ← shared helpers (optional)
│       └── distance.ts
└── dist/
    └── index.mjs                    ← TSUP output, loaded by RealmEngine

Conventions that keep this tidy:

  • One folder per branch under branches/. The folder holds the branch file and its own leaves/ sub-folder. Open branches/combat/ and you see the gate and every action it owns — nothing else.
  • File name = class name. CombatBranch.ts exports CombatBranch. Makes imports unambiguous.
  • One class per file. If two leaves always ship together, one file is fine — otherwise split.
  • Root-level orphans go directly in src/. Leaves that don't belong under any branch (like IdleLeaf) sit at the top of src/, not inside branches/.
  • index.ts is tiny. It just imports and wires the tree. No logic lives here.

  }
}

Open src/branches/combat/ and you see the gate (CombatBranch.ts) and its leaves/ folder side by side — everything combat in one place. TSUP bundles all of it into dist/index.mjs; the folder structure only exists in source.

Status Tracking

While the tree is running, TreeScript tracks the last branch and leaf it stepped through. Read them anywhere (event handler, UI, log message):

RealmEngine.log.info(
  `${this.getCurrentBranchName()} → ${this.getCurrentLeafName()}`,
);

Typical output while actively fighting: CombatBranch → AttackNearest. You get this for free — no this.state = '...' bookkeeping in every leaf.

Customising Branch Behavior

By default, a Branch runs the first valid child. That's what you want most of the time, but when you don't, override onLoop(). A branch that runs every valid child each tick:

import { Branch } from '@realmengine/sdk';

export class RunAllBranch extends Branch {
  isValid(): boolean { return true; }

  onLoop(): number {
    let sleep = 100;
    for (const child of this.getLeaves()) {
      if (!child.isValid()) continue;
      const s = child.onLoop();
      if (s < sleep) sleep = s; // fastest child sets the pace
    }
    return sleep;
  }
}

Heads-up: when you override onLoop(), TreeScript no longer auto-updates getCurrentLeafName() for children of that branch. Call this.setCurrentLeafName(...) from inside your override if you want the status to stay accurate.

Common Pitfalls

  • Expensive isValid(). It may be called on every child every tick. Cache results, don't allocate, don't call network APIs.
  • Forgetting the catch-all leaf. If nothing is valid the tree sleeps for idleSleep (100 ms) and loops at max speed doing nothing. Put an IdleLeaf at the end of the root with a generous sleep.
  • Order of siblings. Remember: first valid child wins. Put safety-critical leaves (heal, nexus) at the top of their branch.
  • Returning 0 from onLoop(). That asks RealmEngine to call you again immediately. Fine briefly, but will spike CPU if it sticks. Return at least a few ms.
  • Not cleaning up. If you subscribe to events in onStart(), unsubscribe in onStop() — exactly like a plain Script.

What's Next?

  • The full API surface (every method on Leaf, Branch, Root, and TreeScript) is in the API Reference tab under TreeScript framework.
  • Skim the API Reference namespaces (self, enemies, world, combat, chat, inventory, …) — those are what your leaves actually call to do work.
  • Stuck? See the Troubleshooting guide, or ask in the RealmEngine Discord.

Creating Your First Plugin

This guide walks you through the simplest possible plugin — one that adds a /hello chat command — then progressively layers on settings, SDK event subscriptions, and types so you have a complete mental model by the end.

A plugin in RealmEngine is a single .mjs file. No folder, no manifest, no build step. Drop the file in, enable it, done. If you've ever written a browser user-script, the shape is similar.

Where Plugins Live

Documents/
└── Realmengine/
    └── Plugins/
        ├── chat-filter.mjs
        ├── damage-log.mjs
        └── auto-greet.mjs
  • The file name (minus .mjs) becomes the plugin id — RealmEngine uses it internally and as the key for saved settings.
  • The display name in the dashboard comes from ctx.name inside your file (next section).
  • Only the top level of the Plugins folder is scanned. Files inside sub-folders are ignored.

Step 1: Your First Plugin

Create Documents/Realmengine/Plugins/hello.mjs and paste:

import { chat } from '@realmengine/sdk';

export function register(ctx) {
  ctx.name = 'Hello';
  ctx.category = 'utility';

  ctx.registerCommand('hello', () => {
    chat.notify('Hi from my first plugin!', 'Hello');
  });
}

Two separate tools are at play here, and the split matters:

  • ctx is the plugin runtime. Tiny. Handles things that only make sense for a single plugin: identity (name, category), dashboard-driven state (enabled), per-plugin settings, per-plugin chat commands. That's the whole surface.
  • @realmengine/sdk is the game API. Everything about the game world — chat, events, enemies, players, self, world, inventory, combat, loot — lives here. Scripts and plugins share the exact same SDK.

So ctx.registerCommand('hello', …) wires the /hello command, and chat.notify('Hi …', 'Hello') (from the SDK) shows the popup. Plugins reach the game exclusively through SDK imports — there is no ctx.chat, ctx.events, or ctx.world.

You can import the SDK in two equivalent styles — pick whichever reads nicer to you:

// Style A — named imports (concise, tree-shakeable).
import { chat, events, RealmEngine } from '@realmengine/sdk';

chat.say('hello');
events.onEnemySpawned(e => {});
const hp = RealmEngine.self.getHp();

// Style B — one big namespace import.
import { RealmEngine } from '@realmengine/sdk';

RealmEngine.chat.say('hello');
RealmEngine.events.onEnemySpawned(e => {});
const hp2 = RealmEngine.self.getHp();

chat and RealmEngine.chat are literally the same object (same for events, inventory, etc.), so the two styles interoperate freely in the same file. After enabling the plugin in RealmEngine:

  1. Open the Plugins tab. Hello should appear within a second or two — plugins are detected automatically.
  2. Toggle the switch on.
  3. In-game, type /hello in chat.
  4. You should see a notification popup in the game client.

How register(ctx) Works

Every plugin exports a single function called register. RealmEngine calls it once when the plugin loads, handing you a context object (we call it ctx) plus access to the SDK via import:

  • Setting ctx.name and ctx.category configures how the plugin appears in the dashboard.
  • Calling ctx.registerCommand attaches a chat command handler. Subscribing to an SDK event (e.g. events.onEnemySpawned) attaches a game-world handler. Both stay attached until the plugin is reloaded.
  • For debug output, just use console.log. It shows up in the RealmEngine process console.

Notice there's no onStart / onLoop / onStop. The body of register() runs once, and after that everything is event-driven — your handlers fire whenever their trigger happens. If you create something that needs tearing down when the plugin reloads (e.g. a setInterval timer, an SDK event subscription), return a cleanup function from register:

export function register(ctx) {
  ctx.name = 'Ticker';

  const handle = setInterval(() => console.log('tick'), 1000);

  return () => {
    clearInterval(handle);
  };
}

RealmEngine calls that returned function once, automatically, whenever the plugin is unloaded, hot-reloaded, or disabled. Anything you created inside register (timers, SDK subscriptions, file handles) should be torn down there.

Step 2: Adding Settings

Settings appear in the dashboard under the plugin's toggle and let users tweak behavior without editing your file:

import { chat } from '@realmengine/sdk';

export function register(ctx) {
  ctx.name = 'Auto Greet';
  ctx.category = 'utility';

  let greeting = 'hi';

  ctx.registerSetting('message', {
    label: 'Greeting message',
    type: 'text',
    value: greeting,
  }, (value) => { greeting = value; });

  ctx.registerCommand('greet', () => {
    chat.notify(greeting, 'Greeter');
  });
}

Walking through the pieces:

  • The first argument ('message') is the setting's key — it's used to save the value between sessions.
  • The second argument is the definition: a label, a type, an initial value, plus any type-specific options (e.g. min/max for range, options for select).
  • The third argument is the onChange callback. It fires whenever the user edits the setting in the dashboard. Capture the value in a variable your hooks can see — here, the greeting closure variable.

Available setting types: text, number, range, boolean, select, button. See the API Reference for each type's options.

Categories

Set ctx.category so your plugin lands in the right section of the Plugins hub:

  • 'combat' — attacks, damage, aim helpers
  • 'movement' — pathing, walk-to, teleport
  • 'automation' — auto-loot, auto-nexus, auto-ability
  • 'visual' — overlays, indicators, HUD
  • 'network' — connections, server switching
  • 'utility' — default; use this if nothing else fits

Step 3: What ctx Gives You (and What It Doesn't)

The ctx object is deliberately tiny. It only covers plugin-runtime concerns the SDK can't scope to an individual plugin on its own — identity, dashboard state, per-plugin settings, per-plugin chat commands. That's it:

  • ctx.name, ctx.category — how you appear in the dashboard.
  • ctx.enabled — read-only boolean reflecting the dashboard toggle.
  • ctx.pluginId, ctx.pluginFile — read-only identity.
  • ctx.registerSetting(key, def, onChange?) — a user-editable setting, as shown above.
  • ctx.getSetting(key) — read a setting's current value.
  • ctx.registerCommand(name, handler) — fires when the user types /name … in chat. Your handler receives the parsed argument list as a string[]; the chat line never reaches the server. Auto-gated by ctx.enabled.

No ctx.log, no ctx.hookPacket, no ctx.world. For debug output use console.log. For cleanup, return a function from register. For anything game-related — chat, events, enemies, world, self, inventory — import it from @realmengine/sdk. You can reach each namespace two equivalent ways:

// Named imports — one line per namespace you use.
import { chat, events, RealmEngine } from '@realmengine/sdk';

chat.notify('example');                    // same as…
RealmEngine.chat.notify('example');        // …this.

events.onEnemySpawned(e => {});            // same as…
RealmEngine.events.onEnemySpawned(e => {}); // …this.

Under the hood they're the same objects — chat === RealmEngine.chat. Use whichever style reads nicer. The full SDK surface is documented in the API Reference tab; the namespaces you'll reach for most are chat, events, RealmEngine.self, RealmEngine.world, RealmEngine.enemies, RealmEngine.players, RealmEngine.inventory, RealmEngine.combat, RealmEngine.walking. A few patterns that come up constantly:

  • React to something happening — use an events.on… subscription. It returns an unsubscribe function; capture it in register and call it from the cleanup you return.
  • Read current state — call a getter on RealmEngine.self, RealmEngine.world, RealmEngine.enemies, RealmEngine.players, RealmEngine.inventory, etc.
  • Talk to the player — chat.notify(message, sender?) for a local-only client message; chat.say(message) to actually send chat.
  • Respect the dashboard toggle — ctx.registerCommand does this automatically, but SDK event handlers don't. If you want them to pause when the plugin is toggled off, check ctx.enabled at the top of the handler.

By design, community plugins do not see raw packets, the internal proxy event bus, the dashboard log panel, or the low-level game-data loader. Those are reserved for bundled first-party plugins the client ships with. If your idea truly needs that level of access (e.g. rewriting packet bytes), ship it as a bundled plugin instead.

Step 4: A Real Example — Spawn Alert

Putting the pieces together. This plugin notifies the player when a specific enemy shows up, with a configurable object ID:

import { chat, events } from '@realmengine/sdk';

export function register(ctx) {
  ctx.name = 'Spawn Alert';
  ctx.category = 'utility';

  let targetId = 2770; // Oryx the Mad God 3 by default

  ctx.registerSetting('objectId', {
    label: 'Enemy object ID to watch',
    type: 'number',
    value: targetId,
  }, (value) => { targetId = value; subscribe(); });

  let unsubscribe = () => {};
  function subscribe() {
    unsubscribe();
    unsubscribe = events.onEnemySpawnedOfType(targetId, (e) => {
      if (!ctx.enabled) return;
      chat.notify(`${e.name} spawned nearby!`, 'Spawn Alert');
      console.log(`[Spawn Alert] ${e.name} (id=${e.objectId}) at (${e.pos.x}, ${e.pos.y})`);
    });
  }

  subscribe();

  return () => {
    unsubscribe();
  };
}

A few details worth calling out:

  • events.onEnemySpawnedOfType returns an unsubscribe function. We hold on to it so we can tear down the old subscription when the setting changes or the plugin reloads.
  • Returning () => unsubscribe() from register tells RealmEngine what to do on reload / disable — no leaked listeners.
  • Checking ctx.enabled inside the handler means the alert respects the dashboard toggle. Skip this check if you want the subscription to keep firing regardless.
  • chat.notify shows a client-side message. It never reaches the server or other players, so it's safe for personal alerts.

Adding Types (Optional)

If you use VS Code or another TypeScript-aware editor, you can get autocomplete on ctx and the SDK without adopting TypeScript. Add one JSDoc comment above register:

/** @param {import('@realmengine/sdk').UserPluginContext} ctx */
export function register(ctx) {
  ctx.name = 'Typed Plugin';
  // ctx methods now autocomplete; SDK imports are typed automatically.
}

For larger plugins, author in TypeScript and bundle with TSUP exactly like a script — see the Packaging With TSUP guide. Your output file just needs to end in .mjs and live in the Plugins folder.

Running Plugins

This is the counterpart to "Running Script Packages", but for plugins — what RealmEngine looks for, how to enable a plugin, and how hot-reload works.

Two Sources

RealmEngine loads plugins from two places, and you'll see both in the Plugins tab:

  • Bundled plugins — first-party plugins that ship with the app (auto-nexus, anti-debuffs, etc.). Always available, always up to date with the client.
  • User plugins — .mjs files you or the community dropped into Documents/Realmengine/Plugins/.

Bundled plugins run with full packet-level access because we ship and trust that code. Community plugins run with a restricted context — they interact with the game exclusively through @realmengine/sdk, the same high-level surface scripts use. Both types register the same way (export a register function, appear as a card in the dashboard), so the authoring experience is nearly identical.

The Plugins Folder

Documents/
└── Realmengine/
    └── Plugins/
        ├── chat-filter.mjs
        ├── damage-log.mjs
        └── my-helper.mjs

The folder is created automatically the first time RealmEngine launches. If for some reason it isn't there, create it yourself — or just drop a .mjs into where it should be and RealmEngine will pick it up on its next scan.

The Plugins Tab

  1. Open RealmEngine and switch to the Plugins tab.
  2. Each plugin shows up as a card with a toggle switch and any settings it registered.
  3. Flip the toggle to enable or disable the plugin. The on/off state is saved between sessions.
  4. Settings appear below the toggle — editing them notifies the plugin immediately.
  5. Bundled plugins (the ones shipped with the client) stream into the Plugin log panel. Community plugins just use console.log for debug output — those lines show up in the RealmEngine process console.

Unlike scripts, you can have as many plugins enabled at the same time as you want — there's no one-at-a-time restriction.

Hot Reload

Edit a .mjs in the Plugins folder and save. RealmEngine detects the change, runs any cleanup functions your plugin registered, and re-executes register() with the new code. Adding a brand-new file works the same way — a new card appears in the tab within a second or two. Deleting a file stops that plugin entirely.

This means you can iterate on a plugin without ever closing RealmEngine. If a change doesn't seem to stick, double-check that the file is directly under Plugins/ (no sub-folders) and that it ends in .mjs.

Rules

  • Only files ending in .mjs are loaded. .js and .cjs files are ignored.
  • The file name (minus extension) is the plugin's stable id. Renaming is treated as "old plugin gone, new plugin appeared" — any saved settings are lost.
  • Each file must export a function called register. Default exports or any other name are skipped with a warning in the log.
  • Community plugins reach the game through the SDK only — no raw packets, no internal proxy events, no game-data internals. That's still plenty of power (chat, movement, auto-actions, webhook spam…). Only install plugins from people you trust.

Troubleshooting

Most first-run failures are small: the wrong folder, a missing field in the manifest, or forgetting to rebuild. Work through the checks below before anything else. Each entry tells you what you'll see, what it means, and how to fix it. If none of them match, hop into the RealmEngine Discord — we're happy to help debug.

Script Doesn't Appear in the Scripts Tab

What you'll see: you press Refresh, but your folder is nowhere in the list.

What it means: RealmEngine didn't recognize your folder as a script package. It's skipping it silently.

Fix — work through these in order, fixing one usually unblocks the rest:

  • Is the folder at Documents/Realmengine/Scripts/<your-folder>? Only that exact path is scanned.
  • Is there a realmengine.script.json file directly inside it? It has to sit next to your source, not inside dist/.
  • Does the manifest have all four required fields — name, developer, version, and entry? A single missing field disqualifies the whole folder.
  • Is the JSON valid? A trailing comma or an unquoted key will silently fail the parse. Paste it into an online JSON validator if you're unsure.

Script Shows an Error Row ("entry not found")

What you'll see: the script appears in the Scripts tab but its status is Error, usually with a message about the entry file.

What it means: RealmEngine found the manifest but the file it points at doesn't exist yet. Almost always because you haven't built — the manifest says dist/index.mjs but there's no dist/ folder.

Fix: run npm run build from your script folder, confirm dist/index.mjs now exists, then press Refresh in the Scripts tab.

SDK Import Fails at Runtime

What you'll see: the script starts, then immediately errors with something like Cannot find module '@realmengine/sdk'.

What it means: TSUP bundled a private copy of the SDK into your output instead of leaving the import alone. The live SDK is provided by RealmEngine at runtime; your bundle should never contain it.

Fix: open tsup.config.ts and make sure the external array contains '@realmengine/sdk'. Rebuild, then run the script again.

Changes Not Running

What you'll see: you rebuilt, but the script still behaves like it did before.

What it means: either the script is still running from the old bundle, or your build failed silently.

Fix:

  • Press Stop in the Scripts tab, then Run again. RealmEngine only re-imports when you restart the run.
  • Check the terminal output of npm run build for errors. Confirm dist/index.mjs's modified timestamp is recent.

Plugin Doesn't Appear in the Plugins Tab

What you'll see: you dropped a .mjs into the Plugins folder but nothing shows up in the dashboard.

What it means: one of the loader's rules wasn't met, so RealmEngine skipped the file.

Fix — check each of these:

  • Is the file at Documents/Realmengine/Plugins/<name>.mjs? Sub-folders are not scanned.
  • Is the extension exactly .mjs? .js and .cjs files are ignored.
  • Does the file export a function called register? Default exports and other names don't count.
  • Look in the main client log for has no register() export, skipping — that confirms the missing export is the issue.

Plugin Threw While Loading

What you'll see: the plugin file exists but its toggle never appears in the Plugins tab.

What it means: RealmEngine called register() and it threw an exception, so the plugin was skipped. Check the main client log for Failed to load plugin <id> and the error message that follows.

Fix — the usual suspects:

  • A syntax error in the file (unbalanced brace, mis-typed keyword, etc.).
  • An exception thrown inside register() — wrap suspect lines in try/catch or comment them out to narrow it down.
  • Importing a module that doesn't exist. Plugins don't have a node_modules/ next to them, so any import needs to come from the SDK or a bundled dependency.

Plugin Edits Not Applying

What you'll see: you saved a change to a .mjs, but the plugin's behavior doesn't update.

What it means: the file watcher didn't see your edit. Possibilities:

  • The file isn't directly under Documents/Realmengine/Plugins/ — the watcher only looks at the top level.
  • The file no longer ends in .mjs.
  • Your editor wrote to a temp file and renamed, which some watchers miss.

Fix: check the plugin log — you should see a "reloaded" line each time you save. If you don't, toggle the plugin off and back on to force a reload. If that works but save-triggered reloads never do, restart RealmEngine once to re-bind the watcher.

Unity Basics

Introduction

RotMG Exalt is built on Unity. You don't need to know Unity to use RealmEngine, but understanding a handful of core concepts makes the SDK — and the game traffic it wraps — much easier to reason about.

This guide is a short primer. It explains the three ideas that show up everywhere in the SDK:

  • Classes — the templates that describe what a thing is (an Oryx, a Sword of Acclaim, a Medusa).
  • Instances — the actual copies of those templates that exist in the world right now.
  • Inheritance — how templates share behavior so that every enemy doesn't have to re-declare how HP or movement works.

If those three ideas click, RealmEngine.world, RealmEngine.enemies, RealmEngine.world.objects, and RealmEngine.world.projectiles all start making sense as different views of the same underlying entities.

Why It Matters for the SDK

The SDK exposes game state in two shapes:

  • Instance data — "this specific Oryx at (123, 456) has 1,200,000 HP left" — comes from live server packets (NEWTICK, ENEMYHIT, UPDATE).
  • Class data — "an Oryx hits for 300 per shot and shoots twelve projectiles" — comes from objects.xml, which the client loads once at startup.

Almost every useful script combines both. Knowing which piece lives where saves hours of debugging.

Terms You'll See

  • GameObject — Unity's name for an instance (an actual thing in the world).
  • Prefab — Unity's name for a class template you can spawn copies of.
  • Component — a piece of behavior or data attached to a GameObject (movement, rendering, stats).
  • ObjectId — the unique network id the server assigns to every spawned instance.
  • Type — the numeric id of the class that an instance was spawned from.

Classes & Prefabs

A class describes a kind of thing — its name, its stats, its behaviors, its sprite. It's a template. A class by itself isn't in the world; it's just a blueprint the server uses when it decides to spawn one.

In Unity these blueprints are called prefabs. In RotMG they live in data/objects.xml as <Object> entries:

<Object type="0x09db" id="Oryx the Mad God 3">
  <Class>Character</Class>
  <MaxHitPoints>2500000</MaxHitPoints>
  <Defense>50</Defense>
  <Projectile id="0">
    <Damage>300</Damage>
    <Speed>80</Speed>
    <LifetimeMS>1500</LifetimeMS>
  </Projectile>
  ...
</Object>

type vs id

Every class has two identifiers:

  • type — a compact number (0x09db). This is what flies across the network. Fast, fixed.
  • id — the human name (Oryx the Mad God 3). This is what you see in the wiki.

The SDK gives you both:

import { RealmEngine } from '@realmengine/sdk';

const oryx = RealmEngine.enemies
  .getAll()
  .find((e) => e.name === 'Oryx the Mad God 3');

if (oryx) {
  console.log(oryx.type);     // 2523  (0x09db)
  console.log(oryx.name);     // "Oryx the Mad God 3"
  console.log(oryx.objectId); // unique per spawn — see "Instances"
}

Classes Don't Have Positions

A class has base HP, base damage, a sprite. It does not have a position, or current HP, or status effects — those belong to the instance. If you find yourself asking "where is the Oryx class?" you probably mean "where is this spawned Oryx?" That's an instance question.

Instances

An instance is a spawned copy of a class. When Oryx's arena loads, the server instantiates one Oryx from the Oryx the Mad God 3 class. When it loads again tomorrow, it instantiates a new one. Same blueprint, different instance.

What's Per-Instance?

Anything that can change while the entity is alive:

  • Position (x, y)
  • Current HP / MP
  • Status effects (stun, paralyzed, etc.)
  • Active projectiles it has fired
  • Whether it's alive at all

These come from live server packets, not objects.xml.

ObjectId Is the Instance Id

The server assigns every spawned instance a unique objectId. It's how packets refer to a specific thing:

// Two Gods of Couscous spawn in the same map. Same class, same type.
// Different objectIds — that's what tells them apart.
const cousci = RealmEngine.enemies
  .getAll()
  .filter((e) => e.name === 'God of Couscous');

cousci.forEach((c) => {
  console.log(c.objectId, c.x.toFixed(1), c.y.toFixed(1), c.hp);
});
// 1243  118.4  72.1  84000
// 1247  120.9  69.8  84000

Instance Lifecycle

server spawns entity → objectId assigned ↓ RealmEngine tracks it in world state (position, HP, …) ↓ server sends updates (NEWTICK, ENEMYHIT) → instance fields change ↓ server removes entity → instance disappears from enemies.getAll()

Because instances come and go, never cache an objectId across map transitions. The same enemy name will get a new id in the next map.

Instance APIs

The SDK namespaces that deal in instances:

  • RealmEngine.enemies — enemy instances on the current map.
  • RealmEngine.players — player instances.
  • RealmEngine.world.objects — non-enemy instances (chests, portals, containers).
  • RealmEngine.world.projectiles — active projectile instances.
  • RealmEngine.self — your own player instance.

Inheritance

Inheritance is how a class reuses behavior from a parent class. In Unity, your custom scripts inherit from MonoBehaviour so they automatically get Update(), Start(), OnDestroy(). In RotMG, classes inherit via the <Class> tag.

The <Class> Tag

Every <Object> in objects.xml has a <Class> child that sets its family. The family defines what kind of entity it is and what fields it's expected to carry.

<Object type="0x09db" id="Oryx the Mad God 3">
  <Class>Character</Class>   <!-- enemy / player family -->
</Object>

<Object type="0x2c16" id="Sword of Acclaim">
  <Class>Equipment</Class>   <!-- gear family -->
</Object>

<Object type="0x0505" id="Greater Potion of Attack">
  <Class>Equipment</Class>   <!-- consumables reuse Equipment -->
</Object>

Common Families

  • Character — enemies and players. Has HP, movement, projectiles.
  • Equipment — weapons, abilities, armor, rings, consumables. Has slot type, stats, sound, activation.
  • Container — chests, loot bags, vault chests.
  • GameObject — decorative or static world objects.
  • Portal — dungeon entrances and exits.
  • Projectile — the bullets themselves are not a top-level class — they're declared inside a Character or Equipment as a <Projectile> child. The parent class owns them.

Two enemies with the same <Class>Character</Class> share the structure (both have HP, both have projectiles) but each can override the values. One Oryx has 2.5M HP, a slime has 100.

What That Means for You

When you pull a class definition from game data, fields are only populated for the family that class belongs to. Asking a Portal for its damage returns nothing — portals aren't characters.

const def = RealmEngine.gameData.getObject(oryxType);
// def.class === 'Character'
// def.maxHP, def.defense, def.projectiles[...]  populated
// def.slotType, def.equipType                    undefined (those are Equipment fields)

If the SDK returns undefined for a field you expected, check the class family first — you might be asking the wrong question of the wrong kind of entity.

Components

Unity builds objects by composition — you attach small single-purpose components to a GameObject. A character might have a Transform (position), a Rigidbody (physics), a Renderer (sprite), and a custom script. Each component handles one thing. Together they make the entity.

Components in RotMG

RotMG follows the same idea. An enemy instance is really a bundle of data slices the server keeps in sync with the client:

  • Transform — position, facing direction. Updated every NEWTICK.
  • Stats — HP, max HP, defense, conditions. Also per-tick.
  • Projectile spawner — the <Projectile> declarations from the class, fired via ENEMYSHOOT.
  • AI / movement — lives on the server; you see the effects, not the logic.
  • Rendering — sprite sheet + texture id; lives entirely client-side.

Components and the SDK

The SDK flattens those components into easy-to-access fields on each instance. You don't fetch a "Transform component" — you just read enemy.x and enemy.y:

const e = RealmEngine.enemies.getAll()[0];

// Transform-ish
e.x; e.y;

// Stats-ish
e.hp; e.maxHp;

// Identity
e.objectId; e.type; e.name;

// Projectile spawner lives on the class, not the instance
const def = RealmEngine.gameData.getObject(e.type);
def.projectiles;  // array of Projectile declarations

The rule of thumb: fields that change moment-to-moment live on the instance; fields that were baked in at design time live on the class.

End-to-End Example

Putting class data and instance data together to answer "how much does Oryx hit for, and where is he right now?":

import { RealmEngine } from '@realmengine/sdk';

const oryx = RealmEngine.enemies
  .getAll()
  .find((e) => e.name === 'Oryx the Mad God 3');

if (oryx) {
  // Instance data — where and how healthy
  console.log(`Oryx at (${oryx.x.toFixed(1)}, ${oryx.y.toFixed(1)})`);
  console.log(`HP: ${oryx.hp} / ${oryx.maxHp}`);

  // Class data — what he's capable of
  const def = RealmEngine.gameData.getObject(oryx.type);
  const shots = def?.projectiles?.length ?? 0;
  const maxDmg = Math.max(...(def?.projectiles ?? []).map((p) => p.damage));
  console.log(`Shoots ${shots} projectile pattern(s), max ${maxDmg} damage per hit`);
}

Rules of Thumb

  • If it can change during a fight, it's on the instance.
  • If it's true of every spawn of this enemy forever, it's on the class.
  • objectId identifies instances; type identifies classes. They are not the same number.
  • Never cache objectId across maps. Cache type or name instead.
  • If an SDK field is undefined, check the class family — you may be reading a Character field off a Portal.

TreeScript framework

A Tree Branch Leaf scaffold for organising scripts as decision trees instead of one giant onLoop. Inspired by DreamBot’s TreeScript. Imported from @realmengine/sdk — no namespace on RealmEngine.

The model

Every node in the tree is a Leaf. A Branch is a Leaf that also holds child leaves. Root is a built-in Branch that is always valid. A TreeScript is a full RealmEngine script whose onLoop walks the tree each tick.

  • Leaf.isValid() — cheap boolean; should this node run right now?
  • Leaf.onLoop() — do the work; return sleep in ms (negative stops the script).
  • Traversal — starting at the root, pick the first child whose isValid() returns true. If it’s a branch, descend; if it’s a leaf, run it and sleep for what it returned.
  • Nothing valid? sleep for TreeScript.idleSleep (default 100 ms) and try again next tick.

Example: combat + vault bot

import { TreeScript, Branch, Leaf, RealmEngine } from '@realmengine/sdk';

class NexusLowHp extends Leaf {
  isValid(): boolean {
    return RealmEngine.self.getHPPercent() < 0.3;
  }
  onLoop(): number {
    RealmEngine.self.escape();
    return 2000;
  }
}

class AttackNearest extends Leaf {
  isValid(): boolean { return RealmEngine.enemies.count() > 0; }
  onLoop(): number {
    const e = RealmEngine.enemies.getNearest();
    if (e) RealmEngine.combat.attack(e.objectId);
    return 150;
  }
}

class CombatBranch extends Branch {
  isValid(): boolean { return RealmEngine.enemies.count() > 0; }
}

class IdleLeaf extends Leaf {
  isValid(): boolean { return true; }
  onLoop(): number { return 500; }
}

export default class MyScript extends TreeScript {
  onStart(): void {
    this.addBranches(
      new CombatBranch().addLeaves(new NexusLowHp(), new AttackNearest()),
      new IdleLeaf(),
    );
  }
}

On every tick the walker enters Root, tries CombatBranch (valid when enemies are present), descends, runs NexusLowHp if HP is critical otherwise AttackNearest. When the map is empty it falls through to IdleLeaf. Plug the current state into the dashboard with getCurrentBranchName() + getCurrentLeafName().
